# Subscriptions Tracker

Java-backend для учёта подписок и регулярных платежей.

Единый реестр обязательств с автоматическим расчётом дат списаний и предупреждениями о скорых платежах. Система поддерживает создание обязательства, оплату, отмену и удаление.

Также реализована возможность просмотреть существующие обязательства с фильтрацией, получить суммы стоимостей, сгруппированные по валютам, и увидеть даты ближайших списаний для рекуррентных подписок в заданном временном диапазоне.


## Стек технологий

- **Java 17** 
- **Spring Boot 4.1** - фреймворк (веб, DI, автоконфигурация)
- **Spring Web (MVC)** - REST API
- **Spring Data JPA + Hibernate** - работа с БД через ORM
- **PostgreSQL** - база данных
- **Flyway** - версионирование схемы БД, миграции применяются автоматически при старте
- **MapStruct** - маппинг Entity <-> DTO: убирает boilerplate, генерирует код на этапе компиляции (без рефлексии, поэтому быстро)
- **Lombok** - генерация геттеров, сеттеров, конструкторов
- **springdoc-openapi** - генерация Swagger/OpenAPI документации
- **JUnit 5 + Mockito** - юнит-тесты
- **H2** - in-memory база для интеграционных тестов; использована вместо Testcontainers, чтобы тесты запускались без Docker у любого проверяющего
- **Docker + Docker Compose** - контейнеризация, запуск всего проекта одной командой


## Запуск

**Предварительные требования:** Docker + Docker Compose. Для локального запуска - JDK 17 (Maven не требуется, используется Maven Wrapper `./mvnw`).

### Запуск через Docker (рекомендуется)

Скопировать шаблон переменных окружения:

​```bash
cp .env.example .env
​```

Собрать и запустить всё одной командой:

​```bash
docker compose up --build
​```

Поднимутся база данных и приложение, миграции Flyway применятся автоматически при старте.

- API: `http://localhost:8080`
- Swagger UI: `http://localhost:8080/docs`

### Локальный запуск (приложение через Maven)

Поднять только базу данных:

​```bash
docker compose up db
​```

Запустить приложение:

​```bash
./mvnw spring-boot:run
​```

## Как запустить тесты

​```bash
./mvnw test
​```

Для тестов не нужен Docker - используется встроенная in-memory база H2, поэтому их можно запускать независимо от того, поднят ли контейнер с PostgreSQL.

Что покрыто:

- **Юнит-тесты** (Mockito, БД замокана): проставление статуса по дате, warning при дубле, сдвиг даты следующего платежа при оплате для каждого типа периодичности, граничные случаи с датами (конец месяца, високосный год), запрет оплаты неактивного обязательства.
- **Интеграционные тесты** (на H2): lazy expiry с исключением для рекуррентных обязательств и фильтрация по категории/статусу - то есть логика, которая живёт в SQL-запросах и не может быть проверена моками.

## Ключевые решения и обоснования

### Lazy expiry (ленивое истечение статуса)

Просроченные обязательства переводятся в статус `expired` не по расписанию (нет фонового scheduler), а **лениво** - непосредственно перед выдачей списка в `GET /obligations`. Перед формированием ответа выполняется bulk-UPDATE, переводящий в `expired` все обязательства со статусом `active`, у которых дата платежа уже прошла (`next_payment_date < today`).

**Важное исключение: рекуррентные обязательства (`recurrence != null`) не истекают.** Логика продуктовая: подписка (например, Netflix) продолжает действовать, даже если пользователь ещё не подтвердил оплату - сервис не должен ошибочно помечать активную подписку как истёкшую. Истекают только разовые обязательства (`recurrence == null`), не оплаченные в срок.

Почему lazy, а не по расписанию: для тестового задания это проще и надёжнее (нет отдельного планировщика, который надо запускать и мониторить), а состояние всегда актуально на момент чтения. `GET /obligations/upcoming` в lazy expiry не нуждается: его окно начинается с `today`, а истечение касается дат строго до сегодняшнего дня, поэтому просроченные в окно не попадают.

### Обработка граничных случаев с датами

При оплате рекуррентного обязательства дата следующего платежа сдвигается на определенный период вперёд. Для этого используются встроенные методы `java.time.LocalDate`: `plusMonths()` и `plusYears()`.

Они корректно обрабатывают конец месяца: например, оплата обязательства с датой **31 января** сдвигает её на **28 февраля** (или 29 в високосный год), а не на несуществующее «31 февраля» - `LocalDate` сам обрезает день до последнего существующего в целевом месяце. Високосные годы также учитываются автоматически.

### Сдвиг даты от плановой даты платежа, а не от даты оплаты

При оплате рекуррентного обязательства новая дата вычисляется относительно **текущего `next_payment_date`**, а не от фактической даты оплаты (`today`).

Пример: если платёж запланирован на 15 августа, а пользователь оплатил его 20 августа (с опозданием), следующая дата станет 15 сентября (15 августа + 1 месяц), а не 20 сентября. Это не даёт съезжать графику платежей при задержках оплаты.

### Интеграционные тесты на H2

Логика lazy expiry и фильтрации находится в SQL-запросах (bulk-UPDATE и JPA Specifications), поэтому проверить её моками нельзя, т.к. мок вернёт заготовленный ответ, не выполняя реальный запрос. Такая логика тестируется на настоящей СУБД.

Изначально для этого использовался Testcontainers (поднимает реальный PostgreSQL в Docker), но на очень свежих версиях Docker возникала несовместимость версий Docker API с клиентом внутри Testcontainers. Чтобы тесты гарантированно запускались одной командой у любого проверяющего без настройки Docker, интеграционные тесты переведены на in-memory базу H2 в режиме совместимости с PostgreSQL. Схему для тестов генерирует Hibernate из сущностей (`ddl-auto: create-drop`), Flyway в тестах отключён.

Компромисс: H2 - эмуляция PostgreSQL, а не он сам, поэтому Postgres-специфичный синтаксис (например, `gen_random_uuid()`) в тестах недоступен. Для проверяемой логики (UPDATE с условиями, фильтры) это не имеет значения. В CI-окружении с предсказуемым Docker правильнее использовать Testcontainers.

### Сброс кэша при bulk-UPDATE (lazy expiry)

Bulk-UPDATE в lazy expiry выполняется напрямую в базе, минуя persistence context (кэш первого уровня Hibernate). Из-за этого объекты, уже загруженные в кэш в рамках текущей транзакции, остались бы со старым статусом, даже если в базе он уже изменён на `expired` - последующее чтение вернуло бы устаревшие данные из кэша.

Поэтому bulk-запрос помечен `@Modifying(clearAutomatically = true)`: после его выполнения persistence context очищается, и следующее чтение идёт в базу за актуальными данными. Эта проблема была выявлена интеграционным тестом (объект оставался `active` после expiry) - наглядный аргумент в пользу интеграционных тестов на реальной БД.

### Динамическая фильтрация через JPA Specifications

`GET /obligations` принимает два опциональных фильтра - `category` и `status`, которые могут комбинироваться в любом сочетании (ни одного, один, оба). Вместо отдельных методов репозитория под каждую комбинацию используются JPA Specifications: каждый фильтр - отдельная спецификация, которая добавляет условие в запрос, только если параметр передан (иначе возвращает `null` и игнорируется). Спецификации комбинируются через `.and()`. Это масштабируемый подход: при добавлении новых фильтров не нужно плодить методы репозитория.

## Примеры запросов

### Создать обязательство
​```bash
curl -X POST http://localhost:8080/obligations \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Netflix",
    "amount": 9.99,
    "currency": "USD",
    "category": "SUBSCRIPTION",
    "recurrence": "MONTHLY",
    "next_payment_date": "2026-08-01"
  }'
​```

### Получить список (с опциональными фильтрами)
​```bash
curl "http://localhost:8080/obligations"
curl "http://localhost:8080/obligations?category=SUBSCRIPTION&status=ACTIVE"
​```

### Ближайшие платежи (окно N дней, по умолчанию 7)
​```bash
curl "http://localhost:8080/obligations/upcoming?days=30"
​```

### Оплатить обязательство
​```bash
curl -X POST http://localhost:8080/obligations/{id}/pay
​```

### Отменить обязательство
​```bash
curl -X PATCH http://localhost:8080/obligations/{id}/cancel
​```

### Удалить обязательство
​```bash
curl -X DELETE http://localhost:8080/obligations/{id}
​```

### Подписаться на события удаления (SSE)
​```bash
curl -N http://localhost:8080/obligations/events
​```

Полная интерактивная документация доступна в Swagger UI: `http://localhost:8080/docs`

## Компромиссы и возможные улучшения

Осознанные упрощения, сделанные с учётом объёма тестового задания:

- **Сообщения об ошибках и warning захардкожены** (вынесены в константы). В продакшене вынесла бы их в `MessageSource` (`messages.properties`) для локализации и изменения без правки кода, а `warning` сделал бы структурированным объектом с машиночитаемым кодом (например, `DUPLICATE_ACTIVE_TITLE`) — чтобы фронтенд и тесты завязывались на стабильный код, а не на текст.

- **Интеграционные тесты на H2, а не Testcontainers.** H2 - эмуляция PostgreSQL; в CI с предсказуемым Docker правильнее Testcontainers на реальном Postgres. Причина выбора - гарантированный запуск тестов без настройки Docker (детали выше).
